OTC Allege Automation — Visual & Conceptual Overview

Single-page reference for business context, rules, stack, architecture, sample inventory, assumptions, and open items. Grounded in the Allege Prototype codebase (React UI + FastAPI pipeline).

1. Business context & scenario clarity

Who initiates the allege?

  • Primary flow: Counterparty (or their ops / confirmations mailbox) emails Nomura to allege or chase a trade, SSI, or confirmation gap.
  • Outbound chase (future): Nomura-initiated follow-up to the counterparty is not the main scenario in this prototype; it is listed as a future enhancement.

What “allege” means (OTC)

An allege is a live dispute or break: the counterparty believes a trade exists (or should be confirmed / settled) in a way that does not align with Nomura’s booking, instructions, or affirmation state. It is distinct from routine confirmations, newsletters, or “issue resolved” follow-ups.

2. Business logic rules

Counterparty inference (priority order)

Implemented in backend/app/services/counterparty.py:

  1. Stated — explicit counterparty in body / extracted counterparty_stated (highest confidence).
  2. Domain-inferred — sender domain maps to a canonical bank via DOMAIN_TO_COUNTERPARTY in backend/app/config.py (only if domain is not a broker domain).
  3. Broker-origin — if sender domain is in BROKER_DOMAINS, the pipeline does not treat the broker as counterparty; counterparty must come from the body / extraction. Otherwise outcome flags low confidence / analyst verification.
  4. Sender-name fallback — unknown domain: best-effort use of display name from headers, with warning-level confidence.
Why broker ≠ counterparty
Inter-dealer brokers (e.g. TP ICAP, Tradition) relay economics between banks; the legal counterparty to Nomura is the bank on the trade, not the broker’s domain. Mapping broker domains to a single bank would be wrong without body-level identification.

Classification scope

backend/app/services/classifier.py — classification uses the latest reply only, not the full thread. Resolved-reply keywords (e.g. “found the trade”, “no further action”) force not an allege.

Exact match logic

  • Systems order: BO → MO → FO CSV reference files; stop at first system that returns at least one candidate (matcher.py).
  • Per product type: key fields differ (FX vs IRS vs CDS, etc.) — see KEY_FIELDS_BY_PRODUCT in matcher.py (trade date, counterparty, pair/currency, notional, rate, direction, value date as applicable).
  • No fuzzy matching in this prototype — normalized string / numeric equality on compared fields.

Low-confidence flag (< 0.6)

Classifier can emit confidence < 0.6. Pipeline still processes the message; the UI surfaces a “Verify classification” style indicator for analysts (see dashboard when aiConfidence < 0.6).

Match vs mismatch vs not-found

Matched
At least one booking row found; compared key fields align with extracted / counterparty line.
Mismatch
Booking found but one or more compared fields differ (detail / entity / rate / SSI path per pipeline rules).
No booking / not found
No row matched across BO→MO→FO under exact rules — UI may show “not found in BO, MO, or FO” or equivalent; distinct from field-level mismatch after a hit.

4. Architecture note

The backend implements a deterministic pipeline — a fixed sequence of steps with explicit branching (classify → extract → resolve CP → match → diff → draft → persist). It is not an autonomous agent: no tool loop, no planner revisiting steps based on emergent sub-goals.

Future agent layer (when it would make sense): ambiguous broker-originated threads, multi-match disambiguation needing clarifying questions, cross-system reconciliation, or adaptive retrieval across many mailboxes could benefit from an agent that orchestrates LLM + tools with human approval gates — still keeping settlement actions human-in-the-loop.

6. Assumptions & rules applied

  • Stop at first hit across BO → MO → FO when searching for booking candidates.
  • Latest reply only drives allege vs not-allege classification.
  • No fuzzy matching on trade keys in the reference matcher (exact normalized compare).
  • Inbound source in UI may show as Email for all file-driven samples (channel abstraction).
  • Case IDs in the demo pipeline use a deterministic hash-style id; production would adopt Nomura’s mailbox / message-id scheme (see open points).

7. Open points & future enhancements

  • Live thread updates — continuous sync of conversation state vs one-shot latest body.
  • Case ID generation — align with Nomura’s production mailbox / Message-ID conventions.
  • Real mailbox ingestion — IMAP / MS Graph / enterprise mail bus instead of file-drop inbox.
  • Real SMTP send — production mail submission vs prototype sent/ folder behaviour.
  • Agent layer — for hard cases (broker ambiguity, multi-match, cross-system chase) under governance.
  • Analyst workflows — e.g. named workflows (e.g. “Arati”) / queue rules and assignment logic beyond demo filters.
  • Real system integration — Murex, DTCC, MarkitWire, SWIFT gateways, etc., instead of CSV stand-ins.
  • Outbound-chase workflow — Nomura-initiated chases as first-class use case.
  • LLM hosting options — Hugging Face endpoints or on-prem models as production alternatives to cloud Claude.
  • Broker list management — admin UI to edit BROKER_DOMAINS / domain maps without code deploy.
